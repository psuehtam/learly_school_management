1. Visao Geral do Sistema
O Learly e um sistema web de gestao completa para escolas de idiomas. Seu objetivo e centralizar, organizar e automatizar os processos academicos, administrativos e financeiros da instituicao, eliminando controles manuais em planilhas, evitando inconsistencias de dados e garantindo historico auditavel de todas as informacoes.

A aplicacao e 100% web, acessada via navegador moderno, sem necessidade de instalacao. A autenticacao e individual por usuario, e o controle de permissoes e baseado em perfis configurados no backend — cada usuario visualiza e executa apenas as acoes autorizadas para sua funcao.

1.1 Principios Fundamentais do Sistema
Regra 1 — Sem exclusao fisica (Soft Delete)
Nenhum dado podera ser excluido fisicamente do banco de dados.
Todo registro que deixar de ser utilizado deve ser marcado com status = 'Inativo', preservando historico completo, rastreabilidade e conformidade legal.
Exemplos: inativar um aluno, um livro, uma turma, um arquivo ou uma parcela — nunca deletar.

Regra 2 — Auditoria total (Audit Log)
Toda acao relevante no sistema gera automaticamente um registro em logs_auditoria.
Cada log contem: data/hora (UTC), usuario, nome do usuario no momento da acao, tabela afetada, id do registro, tipo de acao (INSERT/UPDATE/DELETE/LOGIN/LOGOUT), dados anteriores e novos em JSON.
Os logs sao imutaveis: nunca podem ser alterados ou excluidos.
Implementacao tecnica: Audit.NET integrado ao Entity Framework Core interceptando SaveChanges.

Regra 3 — Multi-tenant isolado
O sistema suporta multiplas escolas na mesma instalacao (multi-tenant).
Todo registro do banco contem escola_id, garantindo isolamento total: dados de uma escola nunca aparecem para outra.
Consultas, relatorios e logs sao sempre filtrados por escola_id.

Regra 4 — Fusos horarios (UTC)
Todos os campos de data/hora sao do tipo DATETIME e gravados em UTC pela aplicacao.
A conversao para o fuso horario local (ex: America/Sao_Paulo) ocorre exclusivamente no frontend.
Motivo: evitar inconsistencias em escolas que possam estar em fusos diferentes (ex: Brasilia UTC-3 vs Manaus UTC-4).

2. Tipos de Usuarios e Perfis de Acesso
O sistema opera com seis perfis de usuario. A partir da v4, o controle de acesso e baseado em Policy-Based Authorization — cada perfil possui um conjunto de permissoes (policies) que controlam acoes especificas, nao apenas acesso a telas. Ver Secao 10 para detalhes.

Perfil	Responsabilidade Principal	Acesso Critico
Professor	Gestao academica das proprias turmas	Chamada, homework, avaliacoes, ocorrencias
Coordenador	Controle pedagogico global e organizacao de turmas	Agenda geral, turmas, matriculas, calendario, books, arquivos, editar template de contrato
Comercial	Gestao do processo comercial e captacao de alunos	Cadastro de pre-alunos, geracao e visualizacao de contratos
Secretaria	Finalizacao de matriculas e gestao administrativa	Finalizar matricula, aprovar matricula, visualizar contratos
Financeiro	Gestao das parcelas e movimentacoes financeiras	Parcelas, historico financeiro, carne escolar
Administrador	Configuracao total do sistema e gestao de escolas	Usuarios, permissoes, todas as telas. Super Admin gerencia escolas.

Nota sobre permissoes granulares
Dentro de cada perfil, as permissoes sao verificadas no backend a cada requisicao. O frontend adapta a interface (oculta botoes, desabilita campos) com base no perfil do JWT, mas a validacao real ocorre na API.
Exemplo: o Professor ve o botao 'Acessar Turma' apenas para suas proprias turmas. O Coordenador ve todas.

3. Controle de Acesso e Usuarios
3.1 Cadastro de Usuarios
O cadastro de novos usuarios e realizado exclusivamente pelo perfil Administrador, acessando o menu Usuarios no painel de configuracoes do sistema.

Botao principal: + Novo Usuario

3.1.1 Campos do Formulario de Cadastro
Campo	Obrigatorio	Observacoes
Nome completo	Sim	Exibido em logs e historico
E-mail	Sim	Serve como login; unico por escola
Senha	Sim	Minimo 8 caracteres, letra maiuscula e numero obrigatorios
Perfil (tipo de usuario)	Sim	Professor / Coordenador / Comercial / Secretaria / Financeiro / Administrador
Permissoes especificas	Sim	Atribuidas por politica (Policy) — configuradas pelo Administrador por usuario
Status	Sim	Ativo (padrao) / Inativo

3.1.2 Regras e Validacoes
•	E-mail deve ser unico dentro da mesma escola (UNIQUE escola_id + email no banco).
•	Usuarios com status Inativo nao conseguem autenticar no sistema — o token JWT nao e gerado.
•	O sistema registra automaticamente data_criacao e data_atualizacao em cada usuario.
•	A senha e armazenada com hash (bcrypt ou equivalente) — nunca em texto plano.
•	Apenas o Administrador pode alterar o perfil de outro usuario.
•	O Administrador nao pode inativar a si proprio.

4. Tela de Login
Tela inicial do sistema, responsavel pela autenticacao do usuario. E a unica tela publica — todas as demais requerem sessao ativa.

4.1 Composicao da Tela
Elemento	Tipo	Comportamento
Codigo da escola	Input text	Campo obrigatorio. Ex: CWB01. Identifica a escola no sistema multi-tenant.
E-mail	Input text	Nao pode estar vazio; validacao de formato de e-mail
Senha	Input password	Icone para exibir/ocultar o texto da senha
Botao Entrar	Button primary	Dispara autenticacao; desabilitado durante carregamento
Mensagem de erro	Alert inline	Exibida abaixo do formulario em caso de falha

4.2 Fluxo de Autenticacao
1.	Usuario informa codigo da escola, e-mail e senha e clica em Entrar.
2.	Backend valida o codigo da escola: localiza escola com status = 'Ativo'. Se invalido, retorna erro generico.
3.	Backend valida credenciais: busca usuario pelo e-mail DENTRO da escola identificada, compara hash da senha, checa status = 'Ativo'.
4.	Em caso de sucesso: busca permissoes (policies) do usuario. Gera JWT com escola_id, usuario_id, perfil_id, lista de permissoes como claims e tempo de expiracao. Registra LOGIN em logs_auditoria.
5.	Frontend armazena o token e redireciona para a tela inicial do perfil correspondente.
6.	Em caso de falha: exibe mensagem clara ('E-mail ou senha invalidos'). Nao revela qual dos dois esta errado.

4.3 Regras de Seguranca
•	Bloqueio temporario apos N tentativas invalidas consecutivas (N configuravel pelo Administrador).
•	O tempo de bloqueio tambem e configuravel (ex: 5 minutos).
•	Todas as tentativas de login (sucesso e falha) sao registradas em logs_auditoria.
•	O token JWT expira apos periodo configuravel; o sistema solicita novo login ao expirar.

5.2.2 Controle de Presenca
Para cada aula registrada na turma, o professor lanca a presenca individualmente por aluno.

Campo da linha de aula	Conteudo
Data da aula	Data real da aula (aulas.data_aula)
Numero da aula	Numero sequencial da aula na turma (ex: Aula 12)
Capitulo planejado	Nome do capitulo vinculado (capitulos.nome)
Conteudo dado	Campo editavel: o professor registra o que foi ensinado (ex: Unit 15, Review 1-3)
Status por aluno	Botoes ou dropdown: P (Presente), F (Falta), FJ (Falta Justificada), R (Reposicao)

Status	Significado	Impacto no calculo de faltas
P — Presente	Aluno compareceu normalmente	Nao conta como falta
F — Falta	Ausencia sem justificativa	Conta como falta
FJ — Falta Justificada	Ausencia com justificativa formal	Conta como falta mas e distinguida em relatorios
R — Reposicao	Aula de reposicao realizada	Abate falta anterior

Regras automaticas:
•	O sistema contabiliza automaticamente o total de faltas por aluno na turma.
•	O historico de presenca e acumulado durante todo o curso — consulta retroativa disponivel.
•	A contagem considera todas as aulas com status 'Realizada' para o calculo percentual de frequencia.

5.2.3 Progresso de Capitulos
Ao finalizar uma aula, o professor pode marcar o capitulo correspondente como concluido diretamente na tela da chamada.

Cenario	Acao do professor	Comportamento do sistema
Capitulo concluido na aula	Clica no botao 'Marcar capitulo como concluido'	Grava concluido=true e data_conclusao em turmas_capitulos_progresso
Capitulo NAO concluido	Finaliza a aula sem marcar conclusao	Sistema exibe opcao: 'Adicionar mais uma aula para este capitulo?'
Professor adiciona aula extra	Confirma a opcao de aula adicional	Nova aula e gerada para o mesmo capitulo, mantendo sequencia numerica

Importante — progresso isolado por turma
O progresso de cada capitulo e registrado individualmente por turma na tabela turmas_capitulos_progresso.
O mesmo capitulo pode estar em andamento em turmas diferentes simultaneamente — o progresso de uma nao afeta a outra.
O card na Agenda (Secao 5.1.3) exibe o capitulo atual buscando o primeiro capitulo nao concluido desta turma.

5.2.4 Controle de Homework
Para cada aula, o professor pode registrar a nota de homework de cada aluno.

Media de homework	Calculada automaticamente	Media aritmetica de todas as notas lancadas do aluno na turma
Historico	Visualizavel	Todas as notas ficam registradas e podem ser consultadas

5.2.5 Avaliacoes
Cada turma possui 5 avaliacoes fixas, vinculadas ao livro. As avaliacoes sao lancadas individualmente por aluno e ficam registradas permanentemente.
Avaliacao	Descricao	Peso / Observacao
Speaking	Avaliacao oral — fluencia e pronuncia	Individual por aluno
Listening	Avaliacao de compreensao auditiva	Individual por aluno
Writing	Avaliacao de producao escrita	Individual por aluno
Class Participation	Participacao e engajamento nas aulas	Individual por aluno
Avaliacao Final	Prova final do livro/nivel	Individual por aluno

•	Notas: decimal de 0 a 10 (ex: 7.5, 9.0).
•	O sistema calcula automaticamente a media das 5 avaliacoes por aluno.
•	O historico de avaliacoes e consultavel por livro e por aluno — mesmo apos remanejamento de turma.
•	Unicidade garantida no banco: UNIQUE (turma_id, aluno_id, tipo_avaliacao) — evita lancamento duplicado.

6.1 Calendario Geral
Permite ao Coordenador gerenciar o calendario letivo da escola, marcando dias que alteram a programacao normal de aulas. O sistema ajusta automaticamente a agenda de todas as turmas e recalcula as datas de termino.

6.1.1 Tela do Calendario
Elemento	Descricao
Visualizacao	Calendario mensal ou anual, com navegacao entre meses
Seletor de tipo de evento	Botoes de acao abaixo do calendario para selecionar o tipo antes de clicar nos dias
Dias marcados	Exibidos com cor de destaque conforme o tipo de evento
Botao Aplicar Alteracoes	Confirma e salva todas as marcacoes realizadas

6.1.2 Tipos de Evento e Impactos
Tipo de Evento	Cor no Calendario	Impacto nas Aulas	Exemplo de uso
AULA	Branco (padrao)	Nenhum — dia normal	Dias letivos normais
SEM AULA	Amarelo	Cancela aulas previstas neste dia	Reuniao de professores, event interno
FERIADO	Vermelho claro	Cancela aulas e justifica ausencia dos alunos	Feriado nacional ou municipal
RECESSO	Azul claro	Cancela aulas por periodo de ferias	Ferias escolares, recesso de julho

6.1.3 Fluxo de Marcacao
7.	Coordenador seleciona o tipo de evento desejado no seletor (ex: FERIADO).
8.	Clica nos dias do calendario para aplicar o evento — cada clique aplica o status e a cor correspondente.
9.	E possivel selecionar multiplos dias consecutivos ou alternados em sequencia rapida.
10.	Clica em 'Aplicar Alteracoes' para confirmar e salvar.

6.1.4 Regras de Impacto no Sistema
•	Turmas com aulas previstas nos dias marcados como SEM AULA, FERIADO ou RECESSO: o sistema ignora automaticamente essas datas no calculo das aulas geradas.
•	A data prevista de termino de todas as turmas afetadas e recalculada automaticamente.
•	Na Agenda do Professor (Secao 5.1), os dias marcados aparecem com destaque de cor, indicando sem aula.
•	Toda inclusao ou alteracao no Calendario e registrada em logs_auditoria com usuario, data, hora e tipo de evento.

Exemplo pratico
Coordenador marca o dia 07/09 como FERIADO.
O sistema verifica todas as turmas que tinham aula naquele dia (segunda-feiras, quartas-feiras, etc.).
As aulas previstas para 07/09 sao automaticamente canceladas.
A data de termino de cada turma afetada e empurrada em 1 aula.
O professor ve o dia 07/09 com fundo vermelho claro em sua agenda.

6.3 Turmas — Ciclo de Vida Completo
O modulo de turmas e o nucleo do planejamento pedagogico. Uma turma representa um grupo de alunos cursando um livro especifico com um professor, em horarios definidos.

6.3.1 Listagem de Turmas
Acesso pelo menu Turmas. Exibe todas as turmas da escola com filtros e busca.

Apos salvar o agendamento, o sistema executa automaticamente:

12.	Nome da turma e gerado definitivamente seguindo o padrao:

Formato: [LIVRO] (IDIOMA) [ABREV LIVRO] [CLASSIFICACAO] / [COD TURMA] [DIA SEMANA] [HORARIO]
Exemplo 1: BOOK 5 (IDIOMA) B5 TEENS / 02 SEG 14:30

O codigo da turma (ex: 02) e sequencial por livro dentro da escola.
13.	Sistema gera automaticamente TODAS as aulas da turma:
•	Para cada semana, cria uma aula em cada dia de turmas_dias_semana.
•	Pula datas marcadas em calendario_geral com suspende_aula = true.
•	Distribui os capitulos conforme qtd_aulas_previstas de cada capitulo.
•	Calcula e grava a data prevista de termino em turmas.data_termino_prevista.

6.3.4 Geracao Automatica de Aulas — Algoritmo
Passo	Descricao
1. Listar datas	A partir de data_inicio, listar todas as datas futuras que correspondem aos dias da semana da turma.
2. Filtrar feriados	Remover datas que existem em calendario_geral com suspende_aula = true para esta escola.
3. Distribuir capitulos	Pegar os capitulos do livro em ordem. Cada capitulo com qtd_aulas_previstas = N recebe N datas consecutivas.
4. Criar registros	Para cada data valida, INSERT em aulas com: turma_id, capitulo_id, numero_aula, data_aula, horario_inicio, horario_fim, professor_id (da turma), status = 'Agendada'.
5. Calcular termino	A ultima data gerada e gravada em turmas.data_termino_prevista.
6. Inicializar progresso	INSERT em turmas_capitulos_progresso para cada capitulo do livro com concluido = false.

6.7.6.2 Regra de Abatimento de Falta
22.	Professor marca presenca como R (Reposicao) na chamada da aula de reposicao.
23.	Sistema registra em presencas: status_presenca = 'R', aula_id = ID da aula de reposicao.
24.	Sistema busca a ultima falta nao compensada do aluno na turma (status_presenca = 'F' sem reposicao vinculada).
25.	Se encontrar: grava reposicao_de_presenca_id na presenca de reposicao apontando para a presenca de falta.
26.	Percentual de frequencia do aluno e recalculado: faltas efetivas = (total F) - (total R vinculadas a F).

Comportamento do abatimento de faltas
A reposicao NAO apaga a falta original - o registro com status='F' permanece intacto.
O calculo de frequencia usa: faltas nao compensadas = COUNT(F sem reposicao vinculada).
Exemplo: aluno com 3 faltas e 1 reposicao realizada = 2 faltas efetivas na frequencia.
O historico exibe ambos: a falta original e a reposicao que a compensou, com rastreabilidade completa.

7.4 Regras de Negocio Financeiro
7.4.1 Integridade no Pagamento de Parcelas
Para alterar o status de uma parcela para 'Pago', os seguintes campos sao OBRIGATORIOS:
•	Data de Pagamento: data real em que o pagamento foi recebido.
•	Valor Pago: valor efetivamente recebido (nao pode ser negativo; nao deve exceder limites configurados).
•	Forma de Pagamento: PIX, Boleto, Cartao, Dinheiro, etc.

7.4.2 Tratamento de Estorno
•	Parcelas com status 'Estornado' ficam completamente bloqueadas para edicao.
•	Permitem apenas visualizacao das informacoes e do historico.
•	Para novo registro referente a parcela estornada: criar nova parcela ou novo recebimento — nunca alterar o registro existente.
•	No banco: o estorno e registrado como nova movimentacao do tipo 'Estorno' em movimentacoes_financeiras — o registro original nunca e alterado.

7.4.3 Rastreabilidade — Log de Alteracoes (parcelas_historico)

Percentual de Desconto	Sim	Percentual anterior, novo percentual

Data de Pagamento	Sim	Data anterior, nova data

Imutabilidade do historico financeiro
Registros em parcelas_historico NUNCA podem ser alterados ou excluidos.

8.2 Detalhamento das Tabelas Novas (v3)
8.2.1 turmas_dias_semana
Finalidade e estrutura
Armazena os dias da semana em que cada turma tem aula como valores inteiros.
Representacao: 0=Domingo, 1=Segunda, 2=Terca, 3=Quarta, 4=Quinta, 5=Sexta, 6=Sabado (padrao JS/Python).
Uma turma que tem aula segunda e quarta tem 2 registros: dia_semana=1 e dia_semana=3.
Permite geracao automatica de aulas, calculo de data de termino e filtragem por dia na agenda.
Restrição: UNIQUE (turma_id, dia_semana) — nao pode duplicar o mesmo dia para a mesma turma.
Indice: idx_turmas_dias_semana (turma_id, dia_semana).

8.2.2 turmas_capitulos_progresso
Finalidade e estrutura
Registra o status de conclusao de cada capitulo DENTRO DO CONTEXTO de cada turma.
O mesmo capitulo pode ter concluido=false em uma turma e concluido=true em outra simultaneamente.
Campos: escola_id, turma_id, capitulo_id, concluido BOOLEAN (padrao false), data_conclusao DATE.
Unicidade: UNIQUE (turma_id, capitulo_id) — um registro por capitulo por turma.

8.5 Auditoria — Implementacao com Audit.NET
8.5.1 Visao Geral
O sistema utiliza Audit.NET integrado ao Entity Framework Core para capturar automaticamente TODAS as operacoes de escrita no banco, sem necessidade de codigo manual em cada service ou repository.

8.5.2 Fluxo de Captura
27.	O DbContext do EF intercepta o metodo SaveChanges antes de qualquer INSERT, UPDATE ou DELETE.
28.	O Audit.NET inspeciona o ChangeTracker e identifica todas as entidades modificadas.
29.	Para cada entidade: captura tipo de acao, tabela afetada, ID do registro, dados anteriores (JSON) e dados novos (JSON).
30.	Injeta escola_id, usuario_id e usuario_nome do contexto do usuario logado (JWT).
31.	Grava o registro em logs_auditoria atomicamente com a operacao principal.

8.5.3 Estrutura do Log
Campo em logs_auditoria	Origem	Exemplo
escola_id	JWT do usuario logado	1
usuario_id	JWT do usuario logado	42 (ou NULL para jobs de sistema)
usuario_nome	Servico de contexto — nome no momento da acao	'Lucas Ferreira'
tabela_afetada	Audit.NET — nome da entidade EF	'parcelas'
registro_id	Audit.NET — PK do registro	'1587'
acao	Audit.NET — tipo da operacao	'UPDATE'
dados_anteriores	Audit.NET — valores antes da alteracao	'{"status":"Pendente","valor_pago":null}'
dados_novos	Audit.NET — valores apos a alteracao	'{"status":"Pago","valor_pago":250.00}'
data_hora	DateTime.UtcNow no momento da captura	'2026-03-19T14:32:11Z'

8.6 Decisoes de Design e Boas Praticas
Topico	Decisao adotada	Justificativa
Exclusao de dados	Soft delete — status = 'Inativo'	Preservar historico, rastreabilidade e conformidade legal
Fuso horario	DATETIME + gravacao em UTC	Evitar inconsistencias entre escolas em fusos diferentes
Charset e ENUMs	utf8mb4; valores ENUM sem acentuacao	Prevenir erros de encoding MySQL em ENUMs
Multi-tenancy	escola_id em TODAS as tabelas	Isolamento total de dados entre escolas na mesma instancia
FK comportamento	ON DELETE RESTRICT (padrao MySQL)	Impede exclusao acidental de registros com dependentes
Polimorfismo em contatos_telefone	Integridade gerenciada pelo back-end	MySQL nao suporta FK em colunas polimorficas
Audit log — campo acao	VARCHAR(30) ao inves de ENUM	Compatibilidade com todos os valores gerados pelo Audit.NET
Historico de parcelas	Tabela parcelas_historico imutavel separada	Rastreabilidade financeira auditavel independente do log geral
Movimentacoes financeiras	Registros imutaveis — estorno cria nova linha	Integridade do fluxo de caixa — nenhuma movimentacao e apagada
Progresso de capitulos	Tabela separada por turma	Mesmo capitulo em estados diferentes em turmas distintas
Dias da semana	Inteiro 0-6 em tabela separada	Permite queries programaticas para geracao automatica de aulas
Horario por aula	TIME em aulas (nao so em turmas)	Permite reagendamento pontual de aulas sem alterar a turma
Autorizacao	Policy-Based com claims JWT	Permissoes granulares por usuario; sem necessidade de deploy para ajustar acesso
Pre-Aluno separado do Aluno	Tabela pre_alunos independente	Preserva funil comercial sem poluir dados academicos
Contrato como template+gerado	Duas tabelas separadas	Template editavel; conteudo gerado e imutavel (preserva o que foi assinado)
Super Admin sem escola_id	Perfil especial sem vinculo a escola	Isola gestao de escolas de dados academicos/financeiros
Login com codigo da escola	Campo adicional no login	Isola autenticacao por escola sem expor estrutura interna

10. Controle de Acesso Avancado — Policy-Based Authorization
A partir da v4, o sistema migra do controle por perfil (role-only) para um modelo de autorizacao baseado em permissoes granulares (Policy-Based Authorization). Isso permite que usuarios do mesmo perfil possam ter conjuntos diferentes de permissoes, e que novas permissoes sejam adicionadas sem alterar o codigo.

10.1 Conceitos
Conceito	Descricao	Exemplo
Role (Perfil)	Agrupamento organizacional do usuario — define contexto geral	Comercial, Secretaria, Coordenador
Policy (Permissao)	Acao especifica que o usuario pode ou nao realizar	CRIAR_PRE_ALUNO, EDITAR_CONTRATO
Claim JWT	Dado embutido no token de autenticacao do usuario	Permissao: CRIAR_PRE_ALUNO
Endpoint protegido	Rota da API que exige uma policy especifica	[Authorize(Policy = 'CRIAR_PRE_ALUNO')]

Por que Policy-Based e nao so Roles?
Com roles puras: um usuario Comercial OU tem acesso a tudo que Comercial acessa, ou nao tem nada.
Com policies: e possivel dar a um usuario especifico apenas um subconjunto de permissoes do seu perfil.
Exemplo: um Comercial trainee pode ter VISUALIZAR_PRE_ALUNO mas nao CRIAR_PRE_ALUNO ate ser liberado.
Isso tambem facilita auditorias: cada acao e associada a uma permissao especifica no log.

No backend: cada endpoint usa [Authorize(Policy='NOME_POLICY')] — o .NET valida o claim no JWT.
No frontend: componentes verificam a lista de permissoes do JWT para exibir/ocultar botoes e telas.

10.5 Super Admin — Gestao de Escolas
O Super Admin e um nivel especial de acesso que existe FORA da estrutura de escolas. Ele nao pertence a nenhuma escola especifica e tem acesso ao painel de administracao global do sistema.

Funcionalidade	Descricao
Criar nova escola	Cadastrar escola com nome, codigo unico de login e status
Criar Admin da escola	Criar o primeiro usuario Administrador de uma escola recem-criada
Gerenciar escolas	Ativar, inativar, editar dados de escolas existentes
Visualizar escolas	Listar todas as escolas com status e data de criacao
Acessar como escola (impersonar)	Entrar no sistema como se fosse um admin da escola para suporte

Super Admin — regras de isolamento
O Super Admin NAO tem acesso a dados academicos ou financeiros de nenhuma escola.
O Super Admin so pode criar/editar escolas e o primeiro Admin de cada escola.
Toda acao do Super Admin e registrada em logs_auditoria com escola_id = NULL.
O login do Super Admin usa um codigo de escola reservado (ex: 'SYSTEM') ou uma rota separada.
Implementacao: usuario com perfil especial no banco (perfil_id = 'superadmin') sem escola_id vinculado.
